## Overview

This system solves a real problem faced by India's grid operators: **electricity cannot be stored, so generation must match demand every second**. As solar and wind generation grows, available capacity changes drastically hour-by-hour — a fixed capacity number is no longer enough.

The system provides four integrated capabilities:

| Capability | What it does |
|---|---|
| **Demand Forecasting** | Predicts next 24h hourly demand for all 5 POSOCO regions simultaneously using Random Forest ML |
| **Dynamic Capacity** | Computes available generation per hour from solar (bell curve), wind (monsoon-driven), hydro (reservoir cycle), and thermal (PLF) |
| **Overload Detection** | Flags every hour where predicted demand exceeds the actual available capacity at that hour |
| **Demand Response** | Recommends 4 India-specific strategies (EV delay, agricultural pump deferral, industrial curtailment, backup generation) with ₹ cost and CO₂ impact |

**Model performance:** All 6 models achieve **> 95% accuracy** and **R² > 0.92** on the held-out test set.

## Quick Start

### 1 · Clone & Dataset

```bash
git clone https://github.com/<your-username>/<your-repo>.git
cd smartgrid
```

The dataset (`India_Hourly_Load.xlsx`) is included in the repository. Place it (or confirm it is) inside `app/data/`:

```
app/
└── data/
    └── India_Hourly_Load.xlsx    ← dataset from GitHub
```

The file must have these columns (exact names):

| Column | Description |
|---|---|
| `datetime` | Hourly timestamp |
| `National Hourly Demand` | All-India MW |
| `Northen Region Hourly Demand` | Northern Region MW *(note: typo in source)* |
| `Western Region Hourly Demand` | Western Region MW |
| `Eastern Region Hourly Demand` | Eastern Region MW |
| `Southern Region Hourly Demand` | Southern Region MW |
| `North-Eastern Region Hourly Demand` | NE Region MW |

> **Note:** The prepare script handles the "Northen" typo automatically.

## Custom Forecast Input

By default, every forecast uses the **last 168 hours (7 days)** from `data/demand.csv`. You can override this with your own recent data directly from the dashboard.

### From the Dashboard UI

1. Click **▼ Enter Custom Historical Data** below the header
2. Set the **Forecast start date/time** (the hour from which the 24h prediction begins)
3. Paste **exactly 168 comma-separated MW values** into any region's text box — oldest first, one value per hour
4. Leave other regions blank to use CSV data automatically
5. Click **⚡ Run Custom Forecast**

**Where to get 168 values:**
- Export the last 7 days of hourly demand from your SLDC/SCADA portal as CSV
- Copy the MW column (168 rows = 7 days × 24 hours)

**Typical MW ranges per region:**

| Region | Min (MW) | Max (MW) |
|---|---|---|
| Northern | 45,000 | 92,000 |
| Western | 55,000 | 105,000 |
| Southern | 38,000 | 78,000 |
| Eastern | 22,000 | 48,000 |
| North-Eastern | 1,200 | 3,800 |
| All-India | 130,000 | 245,000 |

### Via API directly

```bash
curl -X POST http://localhost:8000/api/forecast/all-regions \
  -H "Content-Type: application/json" \
  -d '{
    "Northern_Region_mw": [62400, 61800, 60900, ...],  // 168 values
    "start_datetime": "2024-06-15T00:00:00"
  }'
```

Any region not included in the body falls back to the last 168 rows of `demand.csv` automatically.

## How It Works

### ML Pipeline

```
India Hourly Load XLSX
        ↓
prepare_dataset.py    — auto-detects columns, resamples to hourly, outputs demand.csv
        ↓
preprocessing.py      — builds 21 features per sample
        ↓
train.py              — trains 6 Random Forest models (80/20 chronological split)
        ↓
predictor.py          — recursive 24-step forecast using trained model
```

**21 features** (India-specific):

| Group | Features | Why |
|---|---|---|
| Short lags | `lag_1h`, `lag_2h`, `lag_3h` | Immediate inertia |
| Daily lags | `lag_24h`, `lag_48h`, `lag_72h`, `lag_96h` | Strong daily seasonality |
| Rolling stats | `rolling_7d_mean`, `rolling_7d_max`, `rolling_7d_std` | Weekly patterns |
| Cyclic time | `hour_sin/cos`, `month_sin/cos` | No discontinuity at midnight/Jan |
| Calendar | `is_weekend`, `day_of_week` | Weekend load drop |
| **India peaks** | `is_morning_peak` (7–10h), `is_evening_peak` (18–22h) | India's double-peak profile |
| **India seasons** | `is_summer`, `is_winter`, `is_monsoon` | AC load, irrigation, monsoon shifts |

**Random Forest config:** `n_estimators=100`, `max_depth=12`, `min_samples_leaf=4`, `n_jobs=-1`

---

### Dynamic Capacity Engine

Unlike static capacity models, `capacity_engine.py` computes **available generation per hour** for each region:

| Source | Model | Key pattern |
|---|---|---|
| **Solar** | Gaussian bell curve centred at 12:00 | Zero before 05:00 and after 19:00; monsoon months 40–60% lower |
| **Wind** | Monthly seasonality × regional multiplier | 70% of annual output May–Sep (SW Monsoon); Western (Gujarat) and Southern (Tamil Nadu) highest |
| **Hydro** | Reservoir cycle | Peaks Oct–Nov (full post-monsoon); NE Region peaks Jul–Aug (run-of-river) |
| **Thermal** | Base PLF 58% × seasonal maintenance | Drops to 53% in Mar–May (planned outages); Eastern Region highest (near coalfields) |

**Example:** Northern Region at 02:00 in October → ~62 GW available (thermal only, no solar). At 14:00 in April (solar peak) → ~88 GW. A **26 GW difference** invisible to fixed-capacity models.

---

### Decision Engine

When predicted demand exceeds available capacity, a 4-strategy cascade is applied in order:

```
1. EV Charging Delay     — shift EV loads +2h    (₹600/MWh, zero CO₂)
2. Agricultural Deferral — defer irrigation pumps (₹400/MWh, zero CO₂) ← India-unique
3. Industrial Curtailment — reduce factory loads  (saves ₹2,200/MWh, avoids 820 kg CO₂/MWh)
4. Backup Generation     — gas peakers / DG sets  (₹8,500/MWh, +490 kg CO₂/MWh)
```

Each strategy uses per-hour dynamic capacity (not a single fixed number) and India's actual load segment shares: Industrial 45%, Agriculture 18%, Residential 26%, Commercial 11%.

## Troubleshooting

**`ModuleNotFoundError` when running `uvicorn`**

Make sure you are running from inside the `app/` directory and your virtual environment is activated:
```bash
cd app
source venv/bin/activate   # or venv\Scripts\activate on Windows
uvicorn main:app --reload --port 8000
```

---

**`FileNotFoundError: data/demand.csv not found`**

Run the prepare step first:
```bash
python prepare_dataset.py --file data/India_Hourly_Load.xlsx
```

---

**`No model found for Northern_Region_mw`**

Models need to be trained before the API can forecast. Run:
```bash
python train.py --data data/demand.csv
```

---

**Dashboard shows "Setup Required" banner**

The frontend checks `/api/forecast/status`. Either the backend isn't running, or models aren't trained yet. Check the terminal where `uvicorn` is running for errors.

---

**Frontend can't reach the backend (CORS error)**

The backend allows `http://localhost:3000` by default. If your Next.js dev server runs on a different port, add it to `allow_origins` in `app/main.py` and restart the backend.

---

**Training takes too long**

You can train only the national model first to test the setup, then train regional models:
```bash
python -c "
from preprocessing import preprocess_pipeline
from model import full_training_pipeline
df, X, y, feat = preprocess_pipeline('data/demand.csv', 'demand_mw')
full_training_pipeline(X, y, feat, model_path='model_demand_mw.joblib')
print('National model done')
"
```

---

**`openpyxl` not found**

```bash
pip install openpyxl==3.1.2
```

---

**PyCharm shows red underlines on imports**

Right-click the `app/` folder in PyCharm → **Mark Directory as** → **Sources Root**. This is a PyCharm display issue only and does not affect running the code.
